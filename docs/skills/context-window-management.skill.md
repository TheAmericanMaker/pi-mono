# Context Window Management (Compaction)

## Purpose

Manage LLM context windows by detecting when conversations approach token limits and automatically summarizing older messages while preserving recent context. This allows arbitrarily long agent sessions without hitting context window boundaries, while maintaining awareness of what files were read and modified throughout the session.

## Dependencies

- [Message Transformation Pipeline](./message-transformation-pipeline.skill.md) — message types being compacted
- [Agent Loop with Tool Execution](./agent-loop-with-tool-execution.skill.md) — loop that triggers compaction

## Core Concepts

### Context Window

Every LLM has a maximum number of tokens it can process in a single request (the "context window"). As conversations grow, the total token count approaches this limit. Compaction prevents overflow.

### Token Estimation

Exact token counting requires model-specific tokenizers. A practical approach uses a hybrid strategy: real usage data from the last LLM response (which includes exact token counts) combined with character-based estimation (chars/4) for messages added since.

### Compaction

The process of replacing older messages with a concise summary, reducing token count while preserving essential context. The summary is generated by calling the LLM itself with a specialized summarization prompt.

### File Operation Tracking

Across compactions, track which files were read-only versus modified (written/edited). This information carries forward so the LLM always knows the full file context of the session, even after older messages are summarized away.

### Cut Point

The boundary between messages to summarize and messages to keep. Must be placed at a "clean" boundary — not in the middle of a tool call sequence (between an assistant message with tool calls and its tool results).

## Algorithm / Flow

### Token Estimation (Hybrid Approach)

```
function estimateTokens(message):
    // Simple heuristic: ~4 characters per token
    serialized = serializeMessage(message)
    return len(serialized) / 4

function estimateContextTokens(messages):
    // Find the last assistant message with real API usage data
    lastUsageInfo = null
    for i from len(messages)-1 down to 0:
        msg = messages[i]
        if msg.role == "assistant" and msg.usage and msg.stopReason not in ["aborted", "error"]:
            lastUsageInfo = { usage: msg.usage, index: i }
            break

    if lastUsageInfo is null:
        // No real usage data — estimate everything
        total = 0
        for msg in messages:
            total += estimateTokens(msg)
        return ContextUsageEstimate {
            tokens: total, usageTokens: 0,
            trailingTokens: total, lastUsageIndex: null
        }

    // Use real count up to last usage, estimate the rest
    usageTokens = lastUsageInfo.usage.totalTokens
        or (lastUsageInfo.usage.input + lastUsageInfo.usage.output
            + lastUsageInfo.usage.cacheRead + lastUsageInfo.usage.cacheWrite)

    trailingTokens = 0
    for i from lastUsageInfo.index + 1 to len(messages) - 1:
        trailingTokens += estimateTokens(messages[i])

    return ContextUsageEstimate {
        tokens: usageTokens + trailingTokens,
        usageTokens: usageTokens,
        trailingTokens: trailingTokens,
        lastUsageIndex: lastUsageInfo.index
    }
```

### Should Compact Check

```
function shouldCompact(contextTokens, contextWindow, settings):
    if not settings.enabled:
        return false
    return contextTokens > contextWindow - settings.reserveTokens
```

### Finding the Cut Point

```
function findCutPoint(messages, keepRecentTokens):
    // Walk backwards, accumulating estimated tokens
    accumulated = 0
    cutIndex = len(messages)

    for i from len(messages)-1 down to 0:
        accumulated += estimateTokens(messages[i])
        if accumulated >= keepRecentTokens:
            cutIndex = i
            break

    // Adjust to a clean boundary:
    // Don't cut in the middle of a tool call sequence.
    // A tool call sequence is: assistant message with toolCalls → toolResult messages
    // Walk forward from cutIndex to find a clean boundary.
    while cutIndex < len(messages):
        msg = messages[cutIndex]
        if msg.role == "toolResult":
            cutIndex += 1  // Skip past tool results
        else:
            break  // Clean boundary: user or assistant message (not toolResult)

    // Don't compact everything — keep at least some messages
    if cutIndex <= 1:
        cutIndex = 1
    // Don't keep nothing — leave at least recent messages
    if cutIndex >= len(messages):
        cutIndex = max(1, len(messages) - 2)

    return cutIndex
```

### File Operation Tracking

```
function extractFileOpsFromMessage(message, fileOps):
    if message.role != "assistant": return
    for block in message.content:
        if block.type != "toolCall": continue
        path = block.arguments.path
        if not path: continue

        switch block.name:
            case "read":  fileOps.read.add(path)
            case "write": fileOps.written.add(path)
            case "edit":  fileOps.edited.add(path)

function extractFileOperations(messages, entries, prevCompactionIndex):
    fileOps = { read: set(), written: set(), edited: set() }

    // Carry forward from previous compaction
    if prevCompactionIndex >= 0:
        prevDetails = entries[prevCompactionIndex].details
        if prevDetails:
            for f in prevDetails.readFiles: fileOps.read.add(f)
            for f in prevDetails.modifiedFiles: fileOps.edited.add(f)

    // Extract from current messages
    for msg in messages:
        extractFileOpsFromMessage(msg, fileOps)

    return fileOps

function computeFileLists(fileOps):
    modified = fileOps.edited.union(fileOps.written)
    readOnly = [f for f in fileOps.read if f not in modified].sorted()
    modifiedFiles = [...modified].sorted()
    return { readFiles: readOnly, modifiedFiles: modifiedFiles }
```

### Main Compaction Algorithm

```
function compact(entries, messages, model, settings, extensionHooks):
    // 1. Find cut point
    cutIndex = findCutPoint(messages, settings.keepRecentTokens)

    toSummarize = messages[0:cutIndex]
    toKeep = messages[cutIndex:]

    // 2. Track file operations
    prevCompactionIdx = findLastCompactionEntry(entries)
    fileOps = extractFileOperations(messages, entries, prevCompactionIdx)
    readFiles, modifiedFiles = computeFileLists(fileOps)

    // 3. Allow extensions to prepare (optional hook)
    preparation = {
        messagesToSummarize: toSummarize,
        messagesToKeep: toKeep,
        cutIndex: cutIndex,
        fileOps: fileOps
    }
    extensionHooks.beforeCompact(preparation)

    // 4. Serialize conversation for summarization
    serialized = serializeConversation(toSummarize)

    // 5. Format file operation context
    fileContext = ""
    if readFiles is not empty:
        fileContext += "\n\n<read-files>\n" + readFiles.join("\n") + "\n</read-files>"
    if modifiedFiles is not empty:
        fileContext += "\n\n<modified-files>\n" + modifiedFiles.join("\n") + "\n</modified-files>"

    // 6. Call LLM to generate summary
    summary = callLLM(
        systemPrompt: SUMMARIZATION_SYSTEM_PROMPT,
        userMessage: serialized + fileContext,
        model: model
    )

    // 7. Return result
    return CompactionResult {
        summary: summary,
        firstKeptEntryId: entries[cutIndex].id,
        tokensBefore: estimateContextTokens(messages).tokens,
        details: { readFiles: readFiles, modifiedFiles: modifiedFiles }
    }
```

### Serialization for Summarization

```
TOOL_RESULT_MAX_CHARS = 2000

function serializeConversation(messages):
    parts = []
    for msg in messages:
        if msg.role == "user":
            parts.append("[User]: " + extractText(msg.content))
        elif msg.role == "assistant":
            for block in msg.content:
                if block.type == "text":
                    parts.append("[Assistant]: " + block.text)
                elif block.type == "toolCall":
                    parts.append("[Tool Call: " + block.name + "](" + serialize(block.arguments) + ")")
                // Skip thinking blocks — not relevant for summary
        elif msg.role == "toolResult":
            text = extractText(msg.content)
            text = truncateForSummary(text, TOOL_RESULT_MAX_CHARS)
            prefix = "[Tool Error" if msg.isError else "[Tool Result"
            parts.append(prefix + ": " + msg.toolName + "]: " + text)

    return parts.join("\n\n")

function truncateForSummary(text, maxChars):
    if len(text) <= maxChars: return text
    truncated = len(text) - maxChars
    return text[0:maxChars] + "\n\n[... " + truncated + " more characters truncated]"
```

### Integration with Agent Loop

```
// After each assistant response in the agent loop:
function checkAndCompact(session, agent, model, settings):
    usage = estimateContextTokens(agent.state.messages)

    if shouldCompact(usage.tokens, model.contextWindow, settings):
        entries = session.loadEntries()
        result = compact(entries, agent.state.messages, model, settings, extensionHooks)

        // Persist compaction entry to session
        session.appendEntry({
            type: "compaction",
            summary: result.summary,
            firstKeptEntryId: result.firstKeptEntryId,
            tokensBefore: result.tokensBefore,
            details: result.details
        })

        // Rebuild agent messages from session (summary + kept messages)
        agent.state.messages = rebuildMessagesAfterCompaction(session)
```

### Summarization System Prompt

```
SUMMARIZATION_SYSTEM_PROMPT = """
You are a conversation summarizer. Produce a concise summary of the conversation
that preserves:
1. What the user asked for and the overall goal
2. Key decisions made and their rationale
3. What was accomplished (files created, modified, bugs fixed, etc.)
4. Any pending tasks or unresolved issues
5. Important context that would be needed to continue the conversation

The file lists at the end show ALL files read or modified during the session
(including before any previous compactions). Preserve awareness of these files.

Be concise but thorough. The summary replaces the conversation history,
so anything not in the summary is lost.
"""
```

## Key Design Decisions

1. **Hybrid token estimation** — Uses real API usage data when available (accurate) and falls back to chars/4 heuristic for recent messages. This avoids requiring model-specific tokenizers while staying reasonably accurate.

2. **File operation tracking across compactions** — Each compaction carries forward the cumulative file list from previous compactions. This ensures the LLM always knows what files have been touched in the entire session.

3. **LLM-generated summaries** — Rather than algorithmic summarization, uses the LLM itself to generate summaries. This produces much higher quality summaries that understand the conversation's intent.

4. **Clean cut boundaries** — Never cuts in the middle of a tool call sequence (assistant with tool calls + tool results). This prevents confusing the LLM with orphaned tool results.

5. **Reserve tokens** — Maintains a buffer (reserveTokens) below the context window to ensure the model has room for its response. Without this, the model might receive a full context and have no room to generate output.

6. **Extension hooks** — Extensions can customize compaction behavior (provide their own summaries, inject additional context, track custom state across compactions).

## Edge Cases & Failure Modes

| Scenario | Handling |
|---|---|
| Very short conversation | Don't compact if fewer than 2 messages |
| Conversation is all tool calls | Cut point adjustment walks past tool results to find clean boundary |
| Summary generation fails | Keep original messages; retry compaction next turn |
| Token estimate is very wrong | Reserve tokens provides buffer; worst case, LLM returns context-length error and loop retries |
| Multiple rapid compactions | Each carries forward previous compaction's file lists |
| Messages added during compaction | Compaction is synchronous; new messages wait |
| Empty messages | estimateTokens returns 0; harmless |
| Very large tool results | Truncated to TOOL_RESULT_MAX_CHARS in serialization |
| Extension provides custom compaction | Extension hook can replace the default compaction entirely |
| No previous compaction exists | prevCompactionIndex = -1; skip file list carry-forward |

## Testing Strategy

1. **Unit: Token estimation** — Verify chars/4 heuristic produces reasonable estimates for various message types
2. **Unit: Hybrid estimation** — Messages with real usage data + trailing messages; verify combined estimate
3. **Unit: shouldCompact** — Various token/window/settings combinations
4. **Unit: Cut point finding** — Verify clean boundaries (not between tool call and result)
5. **Unit: Cut point edge cases** — Very short conversations, conversations that are all tool calls
6. **Unit: File operation extraction** — Extract read/write/edit from assistant tool calls
7. **Unit: File list computation** — Files read but not modified vs modified files
8. **Unit: File list carry-forward** — Multiple compactions accumulate file lists correctly
9. **Unit: Serialization** — Conversation serialized correctly with truncated tool results
10. **Unit: Truncation** — Long text truncated with correct character count in marker
11. **Integration: Full compaction** — Generate summary from real messages; verify output is shorter
12. **Integration: Session rebuild** — After compaction, session messages contain summary + kept messages
13. **Integration: Loop integration** — Agent loop triggers compaction when approaching limit; continues working
